# AlgoWiki 发布与更新操作手册

本文档面向当前项目已经验证通过的发布方式：

- 代码仓库：`https://github.com/NullResot/AlgoWiki`
- 镜像仓库：`ghcr.io/nullresot/algowiki-web`
- 服务器目录：`/srv/algowiki`
- 更新方式：GitHub Actions 自动构建镜像，服务器一键拉取并重启

如果你只是要日常发布、更新、回滚，请优先看本文档。

## 1. 标准发布流程

每次发布固定只做 3 步：

1. 本地改代码并推送到 `main`
2. 等 GitHub Actions 的 `Publish GHCR Image` 变绿
3. 服务器执行一条更新命令

服务器日常更新命令：

```bash
cd /srv/algowiki
./deploy/server-update-from-registry.sh
```

### 1.1 这一次先更快上线

如果当前目标是先把这一次发布提速，而不是继续等服务器从 `ghcr.io` 拉镜像，直接在本地运行：

```powershell
cd D:\AlgoWiki_Project
powershell -ExecutionPolicy Bypass -File .\scripts\quick-release-to-server.ps1 `
  -ServerHost 139.224.212.247 `
  -ServerUser root
```

这条链路会：

1. 本地构建镜像
2. 把镜像包和部署辅助文件上传到服务器
3. 服务器执行 `deploy/server-update-from-archive.sh`
4. 直接复用本地构建好的镜像重启容器

适用场景：

- 服务器拉 `ghcr.io` 很慢
- 当前先追求上线速度
- 长期 ACR 方案还没切完

## 2. 第一次使用前检查

第一次切换到 GHCR 更新方式前，确认下面 4 项。

### 2.1 仓库里有 GHCR 工作流和更新脚本

应存在：

- [`.github/workflows/publish-ghcr-image.yml`](../.github/workflows/publish-ghcr-image.yml)
- [`deploy/server-update-from-registry.sh`](../deploy/server-update-from-registry.sh)

### 2.2 GitHub Actions 已成功构建镜像

打开：

- `https://github.com/NullResot/AlgoWiki/actions`

确认 `Publish GHCR Image` 最近一次状态为 `Success`。

构建成功后，通常会生成：

- `ghcr.io/nullresot/algowiki-web:latest`
- `ghcr.io/nullresot/algowiki-web:main`
- `ghcr.io/nullresot/algowiki-web:sha-<commit>`

### 2.3 GHCR package 已设为 Public

GitHub 页面里应能看到：

- package 名称：`algowiki-web`
- 可见性：`Public`

如果不是 `Public`，服务器通常无法直接 `docker pull`。

### 2.4 服务器已存在更新脚本

服务器应存在：

- `/srv/algowiki/deploy/server-update-from-registry.sh`

如果没有，先从本地上传：

```powershell
scp D:\AlgoWiki_Project\deploy\server-update-from-registry.sh root@139.224.212.247:/srv/algowiki/deploy/
```

## 3. 第一次切换到 GHCR

这是只需要做一次的初始化切换。

在服务器执行：

```bash
cd /srv/algowiki
chmod +x deploy/*.sh
./deploy/server-update-from-registry.sh --image ghcr.io/nullresot/algowiki-web:latest --release ghcr-latest
```

脚本会自动：

- 备份 `deploy/.env.production`
- 把 `APP_IMAGE` 改为 `ghcr.io/nullresot/algowiki-web:latest`
- 把 `ALGOWIKI_RELEASE` 改为 `ghcr-latest`
- 拉取最新镜像
- 删除旧的 `algowiki_web_1`
- 重新启动容器
- 尝试执行健康检查

### 3.1 首次切换时的正常现象

如果脚本最后出现：

- `curl: (56) Recv failure: Connection reset by peer`

但此时 `docker ps` 仍显示：

- `Up (health: starting)`

这通常不是失败，而是应用还在启动中。等待 20 到 30 秒后再检查。

检查命令：

```bash
sleep 25
docker ps --format 'table {{.Names}}\t{{.Image}}\t{{.Status}}'
docker logs --tail=120 algowiki_web_1
curl -fsS http://127.0.0.1:8001/api/health/
```

成功标准：

- 容器状态变成 `healthy`
- 日志里出现 `Starting gunicorn`
- `/api/health/` 返回 `{"status":"ok", ...}`

## 4. 日常发布手册

这是后续最常用的部分。

### 4.1 本地推送代码

在本地 PowerShell 执行：

```powershell
cd D:\AlgoWiki_Project
git add .
git commit -m "你的更新说明"
git push origin main
```

如果只是推送已经提交的内容：

```powershell
cd D:\AlgoWiki_Project
git push origin main
```

### 4.2 等待镜像构建完成

打开：

- `https://github.com/NullResot/AlgoWiki/actions`

确认 `Publish GHCR Image` 为绿色成功状态。

不要在 Actions 还没跑完时就去服务器执行更新，否则服务器可能仍拉到旧镜像。

### 4.3 服务器执行更新

在服务器执行：

```bash
cd /srv/algowiki
./deploy/server-update-from-registry.sh
```

### 4.4 发布后检查

更新完成后执行：

```bash
cd /srv/algowiki
docker ps --format 'table {{.Names}}\t{{.Image}}\t{{.Status}}'
curl -fsS http://127.0.0.1:8001/api/health/
docker inspect algowiki_web_1 --format '{{.Config.Image}}'
```

建议顺手检查：

- 首页能否正常打开
- 登录是否正常
- 赛事专区是否正常
- 移动端关键页面是否正常

## 6. 回滚手册

如果最新镜像有问题，不要先手动改服务器源码，优先直接回滚镜像 tag。

```bash

./deploy/server-update-from-registry.sh --image ghcr.io/nullresot/algowiki-web:sha-<commit> --release rollback-sha-<commit>
```

## 7. 常见问题

### 7.1 GitHub Actions 没有推镜像

检查：

- 仓库 `Actions` 是否启用
- `Settings -> Actions -> General -> Workflow permissions` 是否允许 `Read and write permissions`
- `Publish GHCR Image` 工作流是否成功

### 7.2 服务器 `docker pull` 失败

优先检查：

- `algowiki-web` package 是否为 `Public`
- 镜像地址是否正确：`ghcr.io/nullresot/algowiki-web:latest`
- 服务器能否访问 GitHub / GHCR

### 7.3 更新脚本最后健康检查失败

如果脚本最后出现：

- `curl: (56) Recv failure: Connection reset by peer`

先不要直接判定失败，继续执行：

```bash
sleep 25
docker ps --format 'table {{.Names}}\t{{.Image}}\t{{.Status}}'
docker logs --tail=120 algowiki_web_1
curl -fsS http://127.0.0.1:8001/api/health/
```

只要最后状态变成 `healthy` 且健康检查返回 `ok`，就是成功。

### 7.4 再次遇到 `KeyError: 'ContainerConfig'`

你当前服务器还是 `docker-compose 1.29.2`，这个问题是它的老问题。

当前推荐处理方式就是继续使用：

- [`deploy/server-update-from-registry.sh`](../deploy/server-update-from-registry.sh)

因为这个脚本已经先删除旧容器，再执行 compose up，专门绕开了这个问题。

### 7.5 网站更新后还是旧页面

先检查：

```bash
docker inspect algowiki_web_1 --format '{{.Config.Image}}'
grep '^APP_IMAGE=' /srv/algowiki/deploy/.env.production
```

确认容器正在使用的确实是最新 GHCR 镜像。

如果容器镜像已经是最新，但浏览器页面还是旧内容，优先考虑：

- 浏览器缓存
- CDN 缓存
- 前端静态资源缓存
