# 分支与测试环境部署

适用对象：维护测试环境、执行验收发布的开发者。

本文中的域名、仓库和服务器地址均使用占位符。实际部署时替换为自己的值。

本文用 `<PROJECT_ROOT>` 表示本机仓库根目录，用 `<SERVER_PROJECT_DIR>` 表示服务器上的项目目录。两者不需要相同。

## 1. 分支职责

| 分支 | 用途 |
| --- | --- |
| `develop` | 日常开发 |
| `test` | 测试环境发布与人工验收 |
| `main` | 正式环境发布 |

推荐流程：

```bash
git checkout develop
git push origin develop

git checkout test
git merge --no-ff develop
git push origin test

git status  # 测试环境验收通过后
git checkout main
git merge --no-ff test
git push origin main
```

## 2. 镜像规则

GitHub Actions 会为分支构建镜像：

- `ghcr.io/<owner>/algowiki-web:<branch>`
- `ghcr.io/<owner>/algowiki-web:sha-<commit>`
- `<ACR_REGISTRY>/<namespace>/<repository>:<branch>`
- `<ACR_REGISTRY>/<namespace>/<repository>:sha-<commit>`

`latest` 只建议由 `main` 生成。

## 3. ACR 配置

GitHub Actions secrets 示例：

```text
ACR_REGISTRY=<ACR_REGISTRY>
ACR_NAMESPACE=<ACR_NAMESPACE>
ACR_REPOSITORY=<ACR_REPOSITORY>
ACR_USERNAME=<ACR_USERNAME>
ACR_PASSWORD=<ACR_PASSWORD>
```

说明：

- `ACR_PASSWORD` 是 ACR 访问凭证密码，不是云账号登录密码
- 私有 ACR 首次拉取前，需要服务器执行 `docker login`
- 密码只输入到 GitHub secrets 或服务器 Docker 登录，不写入文档

## 4. 测试环境隔离

测试环境应与生产环境隔离：

- Compose project：`algowiki_test`
- 环境文件：`deploy/.env.test`
- 数据库：独立测试库
- 端口：与生产环境不同
- 媒体目录：与生产环境不同
- 镜像标签：跟随 `test` 分支或指定 `sha-*`

## 5. 验证码配置

测试和生产环境建议启用统一验证码。后端运行时环境文件分别配置在 `deploy/.env.test` 和 `deploy/.env.production`，不要把真实密钥提交到 Git。

后端环境变量示例：

```dotenv
CAPTCHA_ENABLED=1
CAPTCHA_REQUIRED_FOR_AUTHENTICATED_USERS=1
TURNSTILE_SITE_KEY=<TURNSTILE_SITE_KEY>
TURNSTILE_SECRET_KEY=<TURNSTILE_SECRET_KEY>
TURNSTILE_VERIFY_URL=https://challenges.cloudflare.com/turnstile/v0/siteverify
SECONDARY_CAPTCHA_ENABLED=0
SECONDARY_CAPTCHA_PROVIDER=geetest
GEETEST_CAPTCHA_ID=<GEETEST_CAPTCHA_ID>
GEETEST_CAPTCHA_KEY=<GEETEST_CAPTCHA_KEY>
GEETEST_VERIFY_URL=<GEETEST_VERIFY_URL>
CAPTCHA_TOKEN_TTL=300
CAPTCHA_FAIL_LOCK_SECONDS=600
REDIS_URL=<REDIS_URL>
```

前端会优先从 `/api/captcha/config/` 读取公开验证码参数，因此测试/生产部署只需要把公开参数和密钥一起写入服务器环境文件。以下 PowerShell 变量仅作为本地构建兜底，一般不需要配置：

```powershell
$env:VITE_TURNSTILE_SITE_KEY="<TURNSTILE_SITE_KEY>"
$env:VITE_SECONDARY_CAPTCHA_ENABLED="0"
$env:VITE_SECONDARY_CAPTCHA_PROVIDER="geetest"
$env:VITE_GEETEST_CAPTCHA_ID="<GEETEST_CAPTCHA_ID>"
```

说明：

- `TURNSTILE_SECRET_KEY` 和 `GEETEST_CAPTCHA_KEY` 是服务端密钥，只放服务器环境文件。
- `TURNSTILE_SITE_KEY` 和 `GEETEST_CAPTCHA_ID` 是公开参数，会由后端配置接口下发给前端。
- 开启二次验证码时，服务器环境文件里的 `SECONDARY_CAPTCHA_ENABLED=1` 即可；前端会从运行时配置同步读取。
- 如果只想先验收 Turnstile，可以保持二次验证码为 `0`。

## 6. 首次初始化

服务器执行。`<SERVER_PROJECT_DIR>` 是服务器上的项目目录，常见值是 `/srv/algowiki`，也可以替换成你自己的目录：

```bash
cd <SERVER_PROJECT_DIR>
git fetch origin
git checkout main
git pull --ff-only origin main
chmod +x deploy/*.sh

./deploy/server-prepare-test-env.sh \
  --domain <TEST_DOMAIN> \
  --image-repo <ACR_REGISTRY>/<ACR_NAMESPACE>/<ACR_REPOSITORY> \
  --github-repo <GITHUB_OWNER>/<GITHUB_REPO> \
  --create-db
```

如果数据库账号没有建库权限，请先手工创建测试库，再不带 `--create-db` 执行脚本。

## 7. 部署测试分支

等待 `test` 分支镜像构建成功后：

```bash
cd <SERVER_PROJECT_DIR>
./deploy/server-update-from-registry.sh \
  --env-file deploy/.env.test \
  --sync-github-branch \
  --github-branch test
```

健康检查：

```bash
curl -fsS -H 'X-Forwarded-Proto: https' http://127.0.0.1:<TEST_PORT>/api/health/
```

## 8. 快部署

如需跳过 Actions 等待，直接把本地当前代码送到测试服务器：

```powershell
cd <PROJECT_ROOT>
powershell -ExecutionPolicy Bypass -File .\scripts\quick-release-test-to-server.ps1 `
  -ServerHost <SERVER_IP_OR_DOMAIN>
```

也可以先设置 `ALGOWIKI_SERVER_HOST` 环境变量，再省略 `-ServerHost`。

快部署只用于测试环境，不用于生产环境。

## 9. Nginx 接入

测试域名反向代理可参考：

- `deploy/nginx.test.algowiki.conf`

如果测试环境直接由 Nginx 提供 HTTPS，需要确保证书、`server_name`、反向代理端口和 Django trusted origins 一致。

## 10. 验收清单

- `/api/health/` 返回成功
- 首页可打开
- 登录、个人中心、动态、审核台正常
- 图片上传与审核链路正常
- 注册发码、找回密码发码、手机号验证发码会弹出验证码
- 学校问卷打开、搜索和草稿自动保存不弹验证码，最终提交会弹验证码
- 管理员可处理动态、评论和举报
- 日志无持续错误
