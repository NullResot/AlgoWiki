# 分支与测试环境部署

本文档用于说明 AlgoWiki 后续的三分支发布流程，以及 `test.algowiki.cn` 测试环境的部署方式。

## 分支职责

- `develop`：日常开发分支。未完成的功能优先提交到这里。
- `test`：测试发布分支。需要内部验证时，把 `develop` 合并到 `test`，GitHub Actions 会构建 `ghcr.io/nullresot/algowiki-web:test` 和对应 `sha-*` 镜像。
- `main`：正式生产分支。只有测试通过的代码才合并到这里，生产环境 `www.algowiki.cn` 从这里上线。

推荐流程：

```bash
git checkout develop
# 开发并提交
git push origin develop

git checkout test
git merge --no-ff develop
git push origin test

# test.algowiki.cn 验证通过后
git checkout main
git merge --no-ff test
git push origin main
```

## 镜像构建规则

`.github/workflows/publish-ghcr-image.yml` 已支持 `main`、`develop`、`test` 三个分支。

分支推送后会生成：

- `ghcr.io/nullresot/algowiki-web:<branch>`
- `ghcr.io/nullresot/algowiki-web:sha-<commit>`
- `ghcr.io/nullresot/algowiki-web:latest`，仅 `main` 生成

部署脚本推荐使用 `--sync-github-branch`，这样服务器会自动解析目标分支最新提交，并部署对应的 `sha-*` 镜像，避免误用旧 tag。

## 测试环境隔离

`test.algowiki.cn` 使用独立环境：

- Compose project：`algowiki_test`
- Docker 网络：`algowiki_test_default`，默认预创建为 `192.168.255.0/24`，避免和 RDS 私网段冲突
- 内部端口：`127.0.0.1:8002`
- 分支：`test`
- 数据库：`algowiki_test`
- 环境文件：`deploy/.env.test`

这样测试容器、Docker 卷和数据库都不会和生产环境混用。

## 服务器首次初始化测试环境

在服务器 `/srv/algowiki` 中执行：

```bash
cd /srv/algowiki
git fetch origin
git checkout main
git pull --ff-only origin main
chmod +x deploy/*.sh

./deploy/server-prepare-test-env.sh --create-db
```

该脚本会同时预创建测试 Docker 网络。若服务器已有同名网络但网段不合适，可先删除测试容器和测试网络，再重新执行脚本：

```bash
docker rm -f algowiki_test_web_1
docker network rm algowiki_test_default
./deploy/server-prepare-test-env.sh --create-db
```

如果数据库账号没有建库权限，先不加 `--create-db`，然后手动创建：

```sql
CREATE DATABASE algowiki_test CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

## 部署测试分支

等待 GitHub Actions 中 `test` 分支的 `Publish GHCR Image` 成功后执行：

```bash
cd /srv/algowiki
./deploy/server-update-from-registry.sh --env-file deploy/.env.test --sync-github-branch --github-branch test
```

健康检查：

```bash
curl -fsS -H 'X-Forwarded-Proto: https' http://127.0.0.1:8002/api/health/
```

## Nginx 接入

将 `test.algowiki.cn` 的 DNS 解析到服务器或当前 CDN 源站后，安装 `deploy/nginx.test.algowiki.conf` 中的反向代理配置。

当前模板是 HTTP 源站反代，适合“公网 HTTPS 由 CDN 终止，CDN 回源到服务器 80 端口”的方式。模板会把 `X-Forwarded-Proto` 设置为 `https`，让 Django 按 HTTPS 站点处理安全 Cookie 和 CSRF。

如果要让 ECS 源站本身也直接提供 `https://test.algowiki.cn`，需要等 DNS 指向服务器后再用 certbot 重新签发覆盖 `test.algowiki.cn` 的证书。

验证：

```bash
nginx -t
systemctl reload nginx
curl -fsS https://test.algowiki.cn/api/health/
```

