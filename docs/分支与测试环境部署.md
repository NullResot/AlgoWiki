# 分支与测试环境部署

本文档用于说明 AlgoWiki 后续的三分支发布流程，以及 `test.algowiki.cn` 测试环境的部署方式。

## 分支职责

- `develop`：日常开发分支。未完成的功能优先提交到这里。
- `test`：测试发布分支。需要内部验证时，把 `develop` 合并到 `test`，GitHub Actions 会构建 GHCR 和 ACR 两套 `test` / `sha-*` 镜像，测试环境默认从 ACR 拉取。
- `main`：正式生产分支。只有测试通过的代码才合并到这里，生产环境 `www.algowiki.cn` 从这里上线。

推荐流程：

```bash
git checkout develop
# 开发并提交
git push origin develop

git checkout test
git merge --no-ff develop
git push origin test

# test.algowiki.cn 验证通过后
git checkout main
git merge --no-ff test
git push origin main
```

## 镜像构建规则

`.github/workflows/publish-ghcr-image.yml` 已支持 `main`、`develop`、`test` 三个分支。

分支推送后会生成：

- `ghcr.io/nullresot/algowiki-web:<branch>`
- `ghcr.io/nullresot/algowiki-web:sha-<commit>`
- `ghcr.io/nullresot/algowiki-web:latest`，仅 `main` 生成
- `crpi-31i8820z65uunzmj.cn-shanghai.personal.cr.aliyuncs.com/algowiki/algowiki-web:<branch>`，当 ACR secrets 配置完整时生成
- `crpi-31i8820z65uunzmj.cn-shanghai.personal.cr.aliyuncs.com/algowiki/algowiki-web:sha-<commit>`，当 ACR secrets 配置完整时生成

部署脚本推荐使用 `--sync-github-branch`，这样服务器会自动解析目标分支最新提交，并部署对应的 `sha-*` 镜像，避免误用旧 tag。

## ACR 配置

测试环境默认镜像仓库：

```text
crpi-31i8820z65uunzmj.cn-shanghai.personal.cr.aliyuncs.com/algowiki/algowiki-web
```

GitHub 仓库需要配置这些 Actions secrets，工作流才会同时推送 ACR：

```text
ACR_REGISTRY=crpi-31i8820z65uunzmj.cn-shanghai.personal.cr.aliyuncs.com
ACR_NAMESPACE=algowiki
ACR_REPOSITORY=algowiki-web
ACR_USERNAME=<ACR 访问凭证登录名>
ACR_PASSWORD=<ACR 固定密码>
```

如果 `ACR_REGISTRY` 已经填成完整仓库路径 `crpi-.../algowiki/algowiki-web`，当前 workflow 也兼容；推荐长期按上面的拆分方式保存。

如果 ACR 仓库不是公开仓库，ECS 首次拉取前还需要在服务器登录一次：

```bash
docker login --username '<ACR 访问凭证登录名>' crpi-31i8820z65uunzmj.cn-shanghai.personal.cr.aliyuncs.com
```

密码使用 ACR 访问凭证里的固定密码，不是阿里云控制台登录密码。

## 测试环境隔离

`test.algowiki.cn` 使用独立环境：

- Compose project：`algowiki_test`
- Docker 网络：`algowiki_test_default`，默认预创建为 `192.168.255.0/24`，避免和 RDS 私网段冲突
- 内部端口：`127.0.0.1:8002`
- 分支：`test`
- 数据库：`algowiki_test`
- 环境文件：`deploy/.env.test`
- 镜像仓库：`crpi-31i8820z65uunzmj.cn-shanghai.personal.cr.aliyuncs.com/algowiki/algowiki-web`

这样测试容器、Docker 卷和数据库都不会和生产环境混用。

## 服务器首次初始化测试环境

在服务器 `/srv/algowiki` 中执行：

```bash
cd /srv/algowiki
git fetch origin
git checkout main
git pull --ff-only origin main
chmod +x deploy/*.sh

./deploy/server-prepare-test-env.sh --create-db
```

该脚本会同时预创建测试 Docker 网络。若服务器已有同名网络但网段不合适，可先删除测试容器和测试网络，再重新执行脚本：

```bash
docker rm -f algowiki_test_web_1
docker network rm algowiki_test_default
./deploy/server-prepare-test-env.sh --create-db
```

如果数据库账号没有建库权限，先不加 `--create-db`，然后手动创建：

```sql
CREATE DATABASE algowiki_test CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

## 部署测试分支

等待 GitHub Actions 中 `test` 分支的 `Publish Container Images` 成功后执行：

```bash
cd /srv/algowiki
./deploy/server-update-from-registry.sh --env-file deploy/.env.test --sync-github-branch --github-branch test
```

健康检查：

```bash
curl -fsS -H 'X-Forwarded-Proto: https' http://127.0.0.1:8002/api/health/
```

## 快部署到测试环境

如果只是想快速把当前本地代码送到 `test.algowiki.cn` 给人工验收，不想等待 Actions 构建和服务器拉镜像，可以直接在本地执行：

```powershell
cd D:\AlgoWiki_Project
powershell -ExecutionPolicy Bypass -File .\scripts\quick-release-test-to-server.ps1
```

这条链路会本地构建镜像，上传镜像包到 ECS，再用 `deploy/.env.test` 重启测试环境。它不会改生产环境。

## Nginx 接入

将 `test.algowiki.cn` 的 DNS 解析到服务器或当前 CDN 源站后，安装 `deploy/nginx.test.algowiki.conf` 中的反向代理配置。

当前模板是 HTTP 源站反代，适合“公网 HTTPS 由 CDN 终止，CDN 回源到服务器 80 端口”的方式。模板会把 `X-Forwarded-Proto` 设置为 `https`，让 Django 按 HTTPS 站点处理安全 Cookie 和 CSRF。

如果要让 ECS 源站本身也直接提供 `https://test.algowiki.cn`，需要等 DNS 指向服务器后再用 certbot 重新签发覆盖 `test.algowiki.cn` 的证书。

验证：

```bash
nginx -t
systemctl reload nginx
curl -fsS https://test.algowiki.cn/api/health/
```
