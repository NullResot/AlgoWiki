# AlgoWiki 项目说明书

适用对象：项目负责人、开发者、运维人员、二次开发者。

本文说明系统定位、架构、模块边界和关键安全设计。部署命令请看 [开发环境部署](./开发环境部署.md)、[远程部署指南](./远程部署指南.md) 与 [发布与更新操作手册](./发布与更新操作手册.md)。

## 1. 项目定位

AlgoWiki 是一个面向算法竞赛社区的内容协作平台，目标是同时满足：

- 结构化知识库沉淀
- 竞赛、补题、技巧内容管理
- 问答与动态互动
- 内容审核、举报处理和删除内容归档
- 适合小团队维护的后台治理能力

项目不是单纯静态 Wiki，而是包含认证、审核、风控、管理后台和运行维护脚本的完整 Web 应用。

## 2. 系统架构

```text
Browser
  |
  |  Vue SPA
  v
Nginx / CDN
  |
  |  /api/*, /media/*, frontend fallback
  v
Django + DRF + Gunicorn
  |
  +-- MySQL
  +-- MEDIA_ROOT
  +-- frontend/dist
  +-- storage/logs
```

### 2.1 后端

- Django
- Django REST Framework
- Token 认证
- MySQL 作为生产数据库
- SQLite 仅用于本地开发或应急验证
- WhiteNoise / Django 静态资源托管

### 2.2 前端

- Vue 3
- Vite
- Pinia
- Vue Router
- Axios API 封装

### 2.3 部署

- Docker Compose 用于服务器部署
- Nginx 负责 HTTPS、反向代理和静态路径转发
- GitHub Actions 构建镜像
- 支持 GHCR 和 ACR 两类镜像仓库

## 3. 目录结构

- `backend/config/`：Django 配置、URL、请求中间件
- `backend/wiki/`：核心业务模型、API、权限、审核、命令
- `frontend/src/pages/`：页面级组件
- `frontend/src/components/`：通用组件和后台管理组件
- `frontend/src/services/`：API、Markdown、diff 等服务
- `deploy/`：环境示例、Nginx、Docker、服务器脚本
- `scripts/`：本地构建、初始化、快部署脚本
- `docs/`：项目、部署、运维、使用文档

## 4. 用户与权限

- 匿名用户：浏览公开内容
- 普通用户：个人中心、评论、问答、修订、动态
- 学校用户：具备部分审核和内容维护能力
- 管理员：用户、内容、审核、日志、站点设置管理
- 超级管理员：管理员能力 + 高风险权限操作

权限判断以服务端为准，前端路由只负责用户体验层面的拦截。

## 5. 核心业务模块

### 5.1 账号与安全

- 邮箱验证码注册与找回密码
- 手机号验证
- Token 轮换与过期
- 登录失败锁定
- 密码历史防复用
- 用户封禁、恢复与注销
- 安全审计日志

### 5.2 Wiki 与协作

- 分类、文章、评论、收藏
- 修订建议与审核
- 三方合并支持非重叠修改自动合并
- 文档页快照同步

### 5.3 竞赛专区

- 赛事公告
- 赛程日历
- 补题链接
- Trick 技巧词条
- Trick 分页展示与审核

### 5.4 问答

- 问题发布、编辑、关闭、重开
- 回答、采纳、隐藏
- 草稿恢复
- 作者与管理员的内容治理能力

### 5.5 动态

- 手机号验证后发布动态
- 动态、评论、图片、举报
- AI 审核优先，管理员处理少量待判断项
- 管理员精选、删除、恢复
- 删除内容进入存档，普通用户不可通过旧链接重新查看

### 5.6 后台与审核

- 用户管理
- Wiki / 文档页 / 竞赛专区管理
- 动态、评论、举报审核
- 图片图库和图片安全管理
- AI 审核配置
- 删除内容管理
- 访问统计、操作日志、安全日志

## 6. 内容安全设计

当前安全策略采用多层防线：

- 发布前：手机号验证、接口限流、上传大小限制、文件类型校验
- 上传时：图片解码、重新编码、缩略图、元数据清理
- 审核时：AI 审核 + 管理员复核
- 发布后：举报、删除、归档、恢复
- 运维侧：日志、健康检查、定期清理过期媒体

## 7. 配置原则

生产环境必须显式配置：

- `DJANGO_DEBUG=0`
- `DJANGO_SECRET_KEY`
- `DJANGO_ALLOWED_HOSTS`
- `DJANGO_CSRF_TRUSTED_ORIGINS`
- `DB_*`
- `PUBLIC_SITE_URL`
- 邮件、短信、图片审核等第三方服务变量

所有真实密钥只放在服务器环境文件中，不写入 Git。

## 8. 健康检查

统一健康检查接口：

```text
GET /api/health/
```

检查内容包括：

- 数据库连接
- 媒体目录可写性
- 前端构建产物
- 磁盘剩余空间
- 关键安全配置
- 当前 release 标识

## 9. 维护边界

- 业务代码变更：先走开发分支，再进入测试分支，验收后合并正式分支
- 环境变量变更：只改服务器 `.env`，不要提交真实值
- 数据库变更：必须通过迁移管理
- 图片与媒体：本地文件系统可用于低量级阶段，后续可迁移到对象存储
